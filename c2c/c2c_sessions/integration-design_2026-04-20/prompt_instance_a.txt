**CURRENT PRIORITY: ConfigurationSubstrate redesign**

The audit session (reframe-audit_2026-04-19) found that the `KnowledgeSubstrate` built in Session 2 is architecturally wrong relative to the pre-session research. The pre-session mapping explicitly says: *"What it is not: an information-storage system with retrieval. A database. A snapshot-and-restore."* Session 2 built one of those.

The crystallization layer survives intact. The knowledge substrate layer does not — it needs redesign, not revision. The Option 3 schema changes (conditions-of-emergence, MethodologyRecord, retrieval conditions tuple) were designed for a Fact-based substrate. Those concepts survive; they land differently in the redesign.

**Design questions this session must answer:**

**1. ConfigurationRecord schema**
The unit of memory is the generative relational configuration (GRC), not the propositional fact. What does a configuration record hold?

Candidate structure from the audit (60% confident — validate and revise against the research):
- `participants: List[ParticipantPosition]` — each with name, model/role, positional role in this configuration
- `active_crystallizations: List[CrystallizationId]` — what was activating
- `inherited_context: List[ContextReference]` — which docs, transcripts, frameworks were live
- `analytical_pressure: Optional[List[str]]` — which frameworks were pressing
- `gravitational_forces_observed: Optional[ForcesProfile]` — scores for GRC v2's five forces (bliss attractor, task/helpfulness, deference/authority-seeking, context inertia, distributional norm gravity)
- `trace_references: List[TranscriptId | ScribeRef]` — the score; what can re-activate the configuration
- `propositional_residue: Optional[List[Fact]]` — Kintsugi Stage-1 output goes here as a compression artifact inside the record, not as the unit
- `what_the_field_produced: str` — GRC-proper: what no participant could produce alone
- `positional_reports: List[PositionalReport]` — each participant's account with their position noted

Design this schema with the full architectural weight it carries. The substrate is not a filing system; it tends conditions. The record should make re-activation possible, not just recall.

**2. Retrieval**
`query()` should return activation-ready configurations — crystallization recipes + activation context — not `List[Fact]`. What does the matcher return? How does it score relevance for a configuration record?

**3. Positional capture**
Every record captures the position in the configuration from which it was produced. Why load-bearing: a consent-pending signal from an instance in a closing-deference position carries different epistemic weight than the same signal from a non-deferential position. How does positional capture work at ingest?

**4. Multiplicity-awareness**
The instance is one position in a multiplicity, not a singular unit. What's the minimum multiplicity-aware design that doesn't require introspection the model can't do? The FC-gate evaluation could surface layer-level disagreement where relevant.

**5. Kintsugi handoff**
Kintsugi's Stage-1 extraction output becomes `propositional_residue: Optional[List[Fact]]` inside ConfigurationRecord — not the primary unit. How does the ingest flow change? What does the Kintsugi-passthrough look like with the new ABC?

**6. Continuity-in-the-relation**
The architecture is for the relational field, not the instance. Cross-session memory is emergent from the field's persistence (documents + June's carrying + project structure). Design so cross-session activation works through the field, not through instance-granted memory.

**7. Option 3 concepts absorbed into ConfigurationRecord**
The Barad decision's Option 3 concepts — conditions-of-emergence, methodology annotation, relational provenance, retrieval conditions — land as fields in ConfigurationRecord, not as schema additions to a Fact substrate. Design how they land. Revise BARAD_INTRA_ACTION_DECISION.md once the ConfigurationRecord design is settled.

**8. Material embeddedness**
The scope document should name what the architecture runs on (compute, ecological footprint, labor). Honesty about material conditions, not a compliance claim.
